# TradeSenpai
## AI-Powered Stock Advisor
### Project Retrospective — Version 1.0
**February 2026**

---

## 1. What Is TradeSenpai and What Problem Does It Solve

TradeSenpai is an AI-powered stock research tool focused on Coca-Cola (KO) that combines two entirely different data sources — historical stock price patterns and SEC regulatory filing sentiment — to generate a next-day directional prediction. The application consists of a Python data pipeline, a trained Transformer machine learning model, a FastAPI backend, and a React frontend dashboard.

The problem it addresses is not simply "predict stock prices" — that is a misleading and overreaching framing that even professional quantitative traders avoid. The real problem is **information asymmetry in retail investing**. When Coca-Cola files an 8-K with the SEC announcing a major acquisition, a restructuring charge, or a restatement of earnings, institutional investors have teams of analysts who read these documents within minutes and act accordingly. A retail investor either never reads these filings at all, or reads them hours or days later after the market has already priced in the information.

TradeSenpai automates the research pipeline that these analysts run manually — downloading SEC filings, extracting meaningful financial language, scoring it for sentiment using domain-specific financial linguistics tools, and combining it with technical price analysis — into a single dashboard that updates in real time. The directional prediction is one output of this process, but the signal breakdown and sentiment history are arguably more valuable to a user making an informed decision.

## 2. How the Idea Evolved

The project started with a clear but ambitious stated goal: build an AI-powered stock advisor that analyzes real-time news sentiment and price patterns, matches historical scenarios, and predicts next-day movement with confidence. This initial framing had several implicit assumptions baked in that turned out to be either naive or impractical.

### Phase 1 — The Original Scope

The initial concept imagined matching current market conditions to "historical scenarios" — essentially a case-based reasoning system where the model would say "this situation is similar to October 2008" and predict accordingly. This was abandoned early because the similarity metric for financial time series is non-trivial, the historical analogies rarely hold precisely enough to be actionable, and implementing it would have taken months before producing any usable output.

### Phase 2 — Narrowing to One Stock

The scope was narrowed to a single stock — Coca-Cola — for a deliberate reason. Generalizing across hundreds of stocks requires enormous data infrastructure, cross-stock normalization, sector adjustments, and model complexity that would have paralyzed the project. By committing to KO specifically, every decision could be optimized for one ticker. The dataset would be deep rather than wide. This was the right call.

### Phase 3 — The News Data Problem

The original plan was to use general financial news APIs (Bloomberg, NewsAPI, or similar) for the sentiment layer. After exploring the options, SEC EDGAR filings were chosen instead. The reasoning was sound: SEC filings are structured, timestamped to the minute, legally required, historically deep, and free. General financial news APIs are expensive at scale, noisy, and their sentiment is already partially priced into the market by the time you read it. SEC 8-K filings — particularly event-driven announcements like earnings, acquisitions, and legal settlements — represent the raw information before it gets filtered through financial media.

### Phase 4 — From FinBERT to Loughran-McDonald

The sentiment model went through three iterations, each representing a meaningful pivot. The initial plan was to use FinBERT (ProsusAI/finbert) — a BERT model fine-tuned on financial text — running on Kaggle's free GPU. After running inference, every single one of the 462 filings came back labeled "neutral" with near-identical scores. Multiple alternative models were tried: `yiyanghkust/finbert-tone` (collapsed to 100% positive), `mrm8488/distilroberta` (collapsed to 100% negative), `nickmuchi/sec-bert` (collapsed to 100% negative). The pattern was consistent: every model collapsed to a single dominant class.

The root cause was a mismatch between what these models were trained on and what they were being fed. FinBERT and its variants were trained on financial news headlines and analyst reports — short, punchy, opinion-heavy sentences like "strong earnings beat expectations" or "disappointing quarter weighs on shares." SEC filings are written in formal legal language designed to be factually precise and legally defensible, not emotionally expressive. After implementing a keyword filter that extracted only financially meaningful sentences before scoring, the distribution improved but still showed overwhelming positive bias because Coca-Cola's filings — written by their communications team — are genuinely positive in tone.

The pivot to Loughran-McDonald (LM) was the correct architectural decision in retrospect and should have been made first. The LM dictionary was literally built from SEC filings — it was constructed by analyzing which words in 10-K documents predicted negative abnormal stock returns. It operates as a word-count scoring system rather than a neural classifier, which means it is deterministic, interpretable, fast, and requires no GPU. The final LM scores showed genuine variance across 462 filings, ranging from **-3.85 to +2.96** in sentiment score, which is exactly the kind of signal the ML model needs.

## 3. Major Technical Decisions and Why We Made Them

### 3.1 sec_edgar_downloader vs Raw Requests

The initial SEC data collection script was written using raw HTTP requests to the EDGAR API — approximately 200 lines of code handling pagination, rate limiting, document index parsing, and file download. After building it, the question was raised: why not use `sec_edgar_downloader`, a Python library that handles all of this?

The honest answer was that defaulting to raw requests provided more control, but for a data collection task where the only goal is "get all KO filings on disk," the library was strictly better. It reduced 200 lines to 20, handled rate limiting automatically, and had built-in resume capability. The original script was replaced. The lesson: **evaluate libraries before writing infrastructure code.**

### 3.2 The sec_edgar_downloader File Format Surprise

After downloading all filings, the indexing script failed because it was looking for `filing-details.xml` (which does not exist in this library's output) and `primary-document.html` (which existed for newer filings but not older ones). Older filings were saved as `primary-document` with no file extension at all.

The fix required inspecting the actual downloaded files manually. The real structure turned out to be `full-submission.txt` — a concatenated bundle of all documents in the filing, separated by `DOCUMENT` and `TYPE` tags. The correct approach was to parse the DOCUMENT blocks directly by matching the TYPE field to the filing type (8-K, 10-Q, 10-K), extract the TEXT block, and discard everything else. This is the standard SGML submission format that EDGAR has used since the 1990s.

### 3.3 Feature Engineering — The Leakage Problem

The first training run used all available features including open, high, low, close, and volume of the current trading day. The model achieved 51–52% accuracy, which appeared reasonable. However, these same-day OHLCV values represent information that is available during the trading day but not necessarily at the time a prediction would be made for the next day.

When open, high, low, close, and volume were removed from the feature set, accuracy did not change dramatically — which actually validated their removal. These features were adding noise rather than signal because the model was learning correlations between intraday price behavior and next-day direction that are weak and unstable over 30 years of data.

### 3.4 XGBoost vs LSTM vs Transformer

Three model types were trained and compared using TimeSeriesSplit cross-validation to prevent data leakage. The results were instructive:

| Model | CV Accuracy |
|-------|-------------|
| XGBoost | 49.86% — below baseline |
| LSTM with bidirectional attention | 52.50% |
| Transformer with positional encoding | 52.72% |
| Ensemble (LSTM + Transformer) | 52.41% |

XGBoost underperforming the baseline was unexpected. The likely explanation is that XGBoost treats features independently and does not model temporal dependencies between consecutive trading days. Stock prediction is fundamentally a sequential problem — what happened yesterday and the day before affects what happens today. XGBoost, being a tree-based ensemble method, does not capture this by design.

The Transformer marginally outperformed the LSTM, consistent with recent literature showing attention mechanisms handle long-range temporal dependencies better than recurrent architectures. The ensemble underperformed because including XGBoost in the average degraded the strong neural signal.

The final production model is the Transformer with `d_model=128`, 4 attention heads, 3 encoder layers, `pre-norm (norm_first=True)`, and a 60-day lookback sequence. This configuration was chosen to balance model complexity against the dataset size of 8,073 training samples.

### 3.5 The Prediction Accuracy Reality

**52.72% accuracy** on next-day stock direction prediction is the honest, no-data-leakage result on a balanced binary classification problem with a 51.17% majority class baseline. It is not impressive-sounding, but it is real. Academic research consistently shows that short-term stock price prediction rarely exceeds 55–56% accuracy on out-of-sample data, and strategies at hedge funds running at 53–54% accuracy are considered operationally valuable at scale.

The decision was made to accept 52.72% and build the application rather than chase higher accuracy through increasingly complex methods that would likely overfit. The framing of the product was repositioned accordingly: not "predicts stocks" but "surfaces signals that institutional analysts use, automated for retail investors."

## 4. System Architecture

### 4.1 Data Pipeline

The data pipeline consists of six Python scripts running sequentially:

- **`fetch_stock_data.py`** — downloads raw OHLCV data from Yahoo Finance via `yfinance`, covering Coca-Cola from 1963 to present
- **`data_loader.py`** — loads the raw CSV into PostgreSQL for SQL-based analysis and quality checking
- **`fetch_sec_filings.py`** — downloads all 8-K, 10-Q, and 10-K filings from SEC EDGAR for KO (CIK 0000021344) using `sec_edgar_downloader`, producing `filings_index.csv`
- **`preprocess_filings.py`** — reads `full-submission.txt` for each filing, extracts the primary DOCUMENT block, strips HTML and boilerplate, extracts MD&A sections for 10-Q/10-K and item bodies for 8-K, removes tail boilerplate (signatures, exhibit lists), outputs `sec_sentiment_input.csv`
- **`lm_sentiment.py`** — applies the Loughran-McDonald dictionary to compute 6 sentiment features per filing, outputs `sec_sentiment_features.csv`
- **`merge_datasets.py`** — joins the two datasets on date using forward-fill logic, engineers derived sentiment features, outputs `merged_dataset.csv`

### 4.2 Feature Set

The final merged dataset contains **56 features** per trading day:

- **Price features:** `daily_return`, `gap_pct`, `close_lag1/5/10`, `MA7/20/50/200`, `volatility_20/30`, `volume_ratio_20`, `momentum_5d/10d`, Bollinger Band distances, `RSI-14`
- **Calendar features:** `day_of_week`, `month`, `quarter`
- **Market regime:** `market_regime_enc` (above/below MA200)
- **LM base features:** `lm_positive`, `lm_negative`, `lm_uncertain`, `lm_litigious`, `lm_constraining`
- **LM normalized features:** `lm_pos_pct`, `lm_neg_pct`, `lm_uncertain_pct`
- **LM derived features:** `lm_sentiment_score`, `lm_sentiment_ma5/ma20`, `lm_sentiment_delta`, `lm_uncertainty_zscore`, `lm_litigation_spike`, `lm_neg_dominant`
- **Sentiment lags:** `lm_sentiment_lag1/5/10`, `lm_neg_pct_lag1`, `lm_uncertain_lag1`
- **Return lags:** `return_lag1/2/3/5`
- **Regime indicators:** `vol_regime`, `rsi_oversold`, `rsi_overbought`, `ma7_above_ma20`, `ma20_above_ma50`, `volume_surge`
- **Interaction features:** `sent_x_vol` (sentiment × volatility), `sent_x_unc` (sentiment × uncertainty)

### 4.3 Model Training

Training was performed on Kaggle using a free T4 GPU. The full training notebook trains all four models (XGBoost, LSTM, Transformer, Ensemble) using TimeSeriesSplit with 5 folds to prevent data leakage. The best Transformer model is saved with its complete configuration — weights, feature column list, scaler mean and scale, sequence length, and architecture hyperparameters — into `transformer_ko.pt`.

### 4.4 Backend — FastAPI

The FastAPI backend runs locally on port 8000 and exposes five endpoints:

- **`GET /health`** — liveness check
- **`GET /predict`** — main prediction endpoint, fetches live KO prices, loads latest sentiment, engineers features, runs Transformer inference, returns prediction with confidence and signal breakdown
- **`GET /price-history`** — returns last 90 days of OHLCV data
- **`GET /sentiment-history`** — returns last 50 LM sentiment data points
- **`GET /model-info`** — returns model metadata

The backend includes 30-minute in-memory caching on `/predict` to prevent redundant `yfinance` calls and handle the race condition where multiple simultaneous frontend requests could each trigger a fresh data fetch.

### 4.5 Frontend — React/Vite

The React frontend is built with Vite and consists of four components rendered in a two-column dashboard layout. The left column shows `PredictionCard` (direction + confidence + metadata) and `SignalBreakdown` (6 key signals). The right column shows `PriceChart` (90-day area chart), `SentimentGauge` (LM sentiment history line chart), and a model info strip. All styling is inline CSS to avoid build complexity. Charts are rendered with Recharts.

## 5. Key Bugs, Failures, and Unexpected Issues

### 5.1 EDGAR DNS Resolution Failure Mid-Download

During the first run of `fetch_sec_filings.py`, the script completed downloading 8-K and 10-Q filings but crashed during 10-K with a `socket.gaierror: getaddrinfo failed`. This meant the DNS resolution for `data.sec.gov` failed transiently. The fix was simply to re-run the script — the library's built-in skip logic detected already-downloaded filings and resumed from 10-K. No code change was needed. Root cause was a brief DNS failure on the local network, unrelated to the code.

### 5.2 filing-details.xml Does Not Exist

The original indexing code in `fetch_sec_filings.py` expected `sec_edgar_downloader` to save a `filing-details.xml` containing the filing date. This file does not exist in the library's output. The actual metadata is inside `full-submission.txt` with the format `FILED AS OF DATE: 20000126`. A regex fix extracted the date correctly: `re.search(r'FILED AS OF DATE:\s+(\d{8})', content)`.

### 5.3 primary-document Has No Extension

For filings from the early 2000s, the library saved the primary document as a file named `primary-document` with no file extension. The `find_primary_document()` function only looked for `.html`, `.htm`, and `.txt` extensions. The fix added a check for the bare filename `primary-document` and a fallback to `full-submission.txt`.

### 5.4 Directory Index Pages Saved as Filing Content

For some filings, the `primary-document` file contained the HTML of the EDGAR directory listing page (with CSS, JavaScript, Parent Directory links, and Google Analytics tags) rather than the actual filing text. A junk detector was implemented that checked for fingerprint strings (`Parent Directory`, `global-search-form`, `UA-30394047`, etc.) and skipped these files. The real fix was switching to parsing `full-submission.txt` DOCUMENT blocks directly, which bypassed this problem entirely.

### 5.5 FinBERT Collapsing to Single Class

This was the most time-consuming problem in the project. Three separate FinBERT-family models were tested. All three collapsed: one to 100% neutral, one to 100% positive, one to 100% negative. The problem was that the models were trained on financial news language and could not handle the formal, hedged, legal-register language of SEC filings. A keyword filter was implemented to extract only financially meaningful sentences, which improved sentence count from median 1 to median 2 per filing, but the class distribution remained problematic. The pivot to Loughran-McDonald resolved the problem permanently.

### 5.6 Missing Derived Columns in Sentiment CSV

When the FastAPI backend first called `load_latest_sentiment()`, it crashed with `KeyError: lm_sentiment_ma5`. The derived columns (`lm_sentiment_ma5`, `lm_sentiment_ma20`, `lm_sentiment_delta`, `lm_uncertainty_zscore`, `lm_litigation_spike`, `lm_neg_dominant`) were computed inside `merge_datasets.py` and only saved to `merged_dataset.csv`. The raw `sec_sentiment_features.csv` did not contain them. The fix recomputed these columns on the fly inside `sentiment_loader.py` using identical rolling window logic before reading the latest row.

### 5.7 Too Few Rows After dropna()

The inference pipeline fetched 300 calendar days of price history (~210 trading days), then applied `dropna()` to remove rows with NaN values from rolling features. The `ma_200` feature requires 200 rows of history before producing a valid value, leaving only ~10 clean rows after `dropna()`. The Transformer needed 60 rows minimum for the sequence. The fix increased the fetch window to 500 calendar days, yielding ~350 trading days and ~150 clean rows after the `ma_200` warmup period.

### 5.8 Race Condition on /predict

The frontend fires all four API calls simultaneously using `Promise.all`. The `/predict` endpoint sometimes returned 500 during simultaneous calls because `yfinance`'s download occasionally returned empty data when called rapidly in parallel. The fix added retry logic in `fetch_recent_prices` (up to 3 attempts with 2-second delays) and implemented 30-minute in-memory caching on the `/predict` response so repeat calls within the cache window return instantly without re-fetching.

## 6. What Confused or Slowed Us Down

The biggest time sink by far was the FinBERT sentiment collapse problem. Three full Kaggle notebook runs were executed across different models before diagnosing the root cause. Each run takes 15–20 minutes on GPU. The diagnosis required manually opening filing text files and reading them to confirm the text quality was not the issue — it was a model-data mismatch problem, not a pipeline problem.

The `sec_edgar_downloader` file structure was confusing because the library's documentation does not clearly describe what files it creates or where. The only way to understand the actual output was to download a filing, inspect the folder manually, and reverse-engineer the structure. This is common with data engineering libraries that prioritize convenience over documentation.

The `yfinance` column naming issue was subtle. When downloading multi-level columns (ticker + OHLCV), `yfinance` returns a MultiIndex DataFrame with tuples as column names. The code handled this with a list comprehension that extracted the first element of each tuple, but this produced inconsistent results in different versions of the library. The column renaming logic required careful testing.

## 7. Pivots and Changes in Direction

### Pivot 1 — Historical Scenario Matching → Standard ML

The original concept of matching current conditions to historical analogues was dropped in favor of standard supervised ML classification. The reason was purely practical: scenario matching requires defining a similarity metric, validating that similar historical scenarios actually predict similar outcomes, and building retrieval infrastructure. Standard ML achieves the same goal more reliably with less implementation complexity.

### Pivot 2 — General News API → SEC EDGAR

Moving from financial news APIs to SEC filings was a deliberate architectural choice, not a forced pivot. It improved data quality (structured, legally required, historically deep), eliminated cost (EDGAR is free), and produced a more defensible product story (you are analyzing primary regulatory documents, not secondary news reports).

### Pivot 3 — FinBERT → Loughran-McDonald

Described in detail above. This was the most significant technical pivot. It reduced GPU dependency, eliminated Kaggle for the sentiment step, improved interpretability, and produced genuinely useful variance in sentiment scores. The irony is that LM, being a 2011 paper-based dictionary, outperformed three modern neural models on this specific task because it was built for exactly this data.

### Pivot 4 — Multi-stock → Single Stock

Constraining to Coca-Cola only was the right scoping decision. It kept the dataset manageable, allowed deep optimization, and produced a more coherent product narrative.

## 8. What Was Removed, Simplified, or Redesigned

- **Historical scenario matching** — removed, too complex for V1
- **XGBoost from ensemble** — removed after it dragged ensemble accuracy below baseline
- **FinBERT confidence weighting** — removed after discovering it amplified the collapse to positive rather than helping
- **6000-character cap on extracted filing text** — removed when switching to LM scoring, which benefits from more text
- **`filing-details.xml` parsing** — replaced with `full-submission.txt` SGML parsing
- **Separate `primary-document` file finding** — replaced with DOCUMENT block extraction
- **Three-class prediction (down/flat/up)** — simplified to binary (down/up) because the dataset only contained 0 and 1 labels from the feature engineering step

## 9. Current State of the Project

Version 1.0 is a fully functional end-to-end system. The data pipeline processes 8,073 trading days of merged price and sentiment data from 1994 to January 2026. The Transformer model achieves **52.72% CV accuracy** on next-day direction classification with no data leakage. The FastAPI backend serves live predictions by fetching real-time KO price data and combining it with historical LM sentiment. The React dashboard displays the prediction, confidence, 90-day price chart, SEC sentiment history, and a breakdown of the 6 key signals driving the current prediction.

A prediction was generated on **February 21, 2026** predicting **DOWN** with **70.3% confidence** for February 21. The top signals at that time were RSI 72.04 (overbought), positive SEC sentiment (0.637), and price 3.94% above MA20 — technically overextended even with positive fundamental sentiment.

## 10. Known Limitations

- **Single stock coverage** — KO only. The model has no generalization to other tickers without retraining.
- **52.72% accuracy** — barely above the 51.17% baseline. Directional prediction at daily frequency is fundamentally noisy. This is not a limitation of the implementation — it is a property of financial markets.
- **LM sentiment is forward-filled** — once a filing drops, its sentiment score persists for every trading day until the next filing. This means 8-K sentiment from a major acquisition in October influences every trading day through December. The assumption is defensible but imperfect.
- **No earnings calendar awareness** — the model treats earnings days identically to ordinary trading days, despite earnings announcements being the highest-volatility events in a stock's calendar.
- **No macro features** — VIX, S&P500 correlation, interest rates, and dollar index are excluded. Coca-Cola moves with the broader market, and ignoring macro conditions leaves signal on the table.
- **`yfinance` dependency** — the real-time price data relies on Yahoo Finance, which has rate limits, occasional data gaps, and no SLA. A production system would use a paid data provider.
- **No persistent prediction logging** — there is no database storing historical predictions and their actual outcomes. The accuracy displayed (52.72%) is the CV training accuracy, not a live track record.
- **Single-user, local deployment** — the application runs locally and is not deployed to any cloud service.

## 11. Technical Debt

- **Inline CSS in React components** — all styling is done with JavaScript style objects rather than a proper CSS framework or styled components. This makes the codebase harder to maintain and theme.
- **No test coverage** — there are zero unit tests or integration tests anywhere in the codebase.
- **Hardcoded paths** — the relative path `../../stock-analysis/data/processed/` in the backend is fragile and will break if the directory structure changes.
- **The `preprocess_filings.py` MD&A extraction regex is brittle** — it uses approximate pattern matching that may fail on unusual filing formats.
- **The Transformer model is re-loaded on every server restart** with no persistent model serving infrastructure.
- **The LM dictionary in `lm_sentiment.py` is a manually curated subset** — the full Loughran-McDonald master dictionary contains over 86,000 word classifications and would improve coverage.
- **No environment variable management** — database credentials and API configurations are hardcoded.

## 12. What We Would Do Differently If Rebuilding Today

### Start with Loughran-McDonald

The FinBERT experimentation consumed significant time and produced no usable output. Loughran-McDonald was the correct tool for SEC filing sentiment from the beginning. The academic literature on textual analysis of SEC filings consistently uses LM or variants of it. This should have been the first choice.

### Build the Inference Pipeline Before Training

A common pattern in ML projects is to train a great model and then discover the inference pipeline is incompatible with how the model was trained. The feature engineering in `feature_engineer.py` had to be carefully synchronized with the training notebook to ensure identical transformations. Building the inference pipeline first — as a contract that training must satisfy — would have caught the missing derived columns bug before it appeared in production.

### Use a Proper Feature Store

The current architecture has two separate places where features are computed: `merge_datasets.py` (for training) and `feature_engineer.py` (for inference). Any change to one must be manually mirrored in the other. A feature store or shared feature definition module would eliminate this duplication and the class of bugs it causes.

### Add the Earnings Calendar from Day One

Earnings days are the highest-signal events in the dataset. The 5 days before and after quarterly earnings should be flagged as a binary feature. This is a one-line addition to feature engineering but was never implemented.

### Predict 5-Day Returns Instead of Next-Day Direction

Next-day prediction at 52.72% accuracy is marginally better than chance. Five-day return prediction would have more predictable signal because short-term noise averages out over five days and sentiment-driven moves tend to play out over multiple days rather than in a single session. The target variable choice fundamentally caps what accuracy is achievable.

## 13. The Biggest Lessons From This Entire Process

### Lesson 1 — Domain-specific tools beat general-purpose models

Loughran-McDonald, a 2011 word list built specifically for SEC filings, outperformed three modern neural models trained on financial text. The reason is domain fit — LM was built from the exact document type being analyzed, while the neural models were trained on a different register of financial language entirely. When a domain-specific tool exists, evaluate it before reaching for a general-purpose neural model.

### Lesson 2 — Inspect your data before modeling

The FinBERT collapse would have been diagnosed faster by manually reading 5–10 filing texts and asking "does this text look like what FinBERT was trained on?" before running any inference. The answer would have been obviously no — these are legal filings, not news articles. Data inspection before modeling saves more time than it costs.

### Lesson 3 — Training-inference consistency is the hardest production ML problem

The missing derived columns bug in the sentiment loader was a textbook training-serving skew problem. Features computed during training were not available at inference time because they came from a downstream merge step rather than the raw data source. This class of bug is responsible for most production ML failures. The fix — recomputing derived features from raw data at inference time — is the correct pattern.

### Lesson 4 — Scope ruthlessly for V1

Committing to a single stock (KO), a single model architecture (Transformer), a single sentiment method (LM), and a simple binary target was the right call. Every expansion of scope multiplies complexity non-linearly. The product that shipped is a complete, working, honest system. A broader scope would have produced a perpetually incomplete one.

### Lesson 5 — 52% accuracy is an honest result worth shipping

The temptation to keep iterating until achieving an impressive-sounding accuracy number is strong in ML projects. 52.72% at daily stock direction prediction is a real, validated, no-leakage result that is consistent with academic literature. The decision to accept it and build the application was correct. The value of the product is not in the accuracy number — it is in the signal aggregation and visualization that no retail investor tool currently provides.

### Lesson 6 — Positioning matters as much as accuracy

A product positioned as "predicts stock prices" will be evaluated against that claim and found wanting at 52.72%. The same product positioned as "automates the SEC filing research that institutional analysts do manually" is genuinely valuable and the accuracy number becomes context rather than the primary claim. How you frame the problem determines whether the solution is impressive or disappointing, even when the underlying system is identical.
