## 1. What TradeSenpai Is

An AI-powered stock research platform. Not a trading system — a research augmentation tool that surfaces institutional-grade signals for retail investors. Core philosophy: domain-specific beats general-purpose, honest beats impressive.

**Live URLs:**
- Frontend: https://trade-senpaii.vercel.app

## 5. Backend Architecture

### 5.1 Startup Sequence (`main.py` lifespan)
```
1. Preload all 6 PyTorch models into memory (prevents cold start lag)
2. Initialize Supabase connection via alert_store.py
3. Start APScheduler (morning brief, evening brief, weekly digest, signal watcher)
4. Start Telegram bot listener in daemon thread (auto-approve subscribers)
5. FastAPI begins serving requests
```

### 5.2 All Endpoints

| Method | Path | Auth | Rate Limit | Description |
|--------|------|------|-----------|-------------|
| GET | /health | None | None | Status, version, tickers |
| GET | /tickers | None | None | Ticker list with names |
| GET | /predict | None | None | Transformer prediction |
| GET | /price-history | None | None | 200d OHLCV data |
| GET | /sentiment-history | None | None | LM sentiment per filing |
| GET | /model-info | None | None | CV accuracy, features |
| GET | /explain | None | 10/min | LLM explanation + analogies |
| POST | /hypothesis/stream | None | 5/min | 6-agent SSE stream |
| POST | /subscribe | None | 3/min | Telegram subscription |
| GET | /subscribers | JWT Admin | None | List all subscribers |
| POST | /subscribers/{id}/approve | JWT Admin | None | Approve subscriber |
| POST | /subscribers/{id}/reject | JWT Admin | None | Reject subscriber |

### 5.3 ML Model Architecture
```
Input:  60-day sequence × 56 features
Model:  Transformer encoder
        - d_model=128, heads=4, layers=3
        - dropout=0.3, norm_first=True (pre-norm)
        - Classifier: Linear(128→64) → GELU → Dropout → Linear(64→2)
Output: Binary classification (UP/DOWN) + confidence score
Note:   torch.load() requires weights_only=False (non-tensor metadata in checkpoints)
```

### 5.4 56-Feature Groups
- Price features: returns, gaps, MAs (7/20/50/200), volatility, RSI, Bollinger
- Calendar: day of week, month, quarter
- Regime: binary above/below MA200
- LM base: positive, negative, uncertain, litigious, constraining counts
- LM normalized + derived: sentiment score, deltas, z-scores, spike flags
- Sentiment lags: 1/5/10 day
- Return lags: 1/2/3/5 day
- Interaction terms: sentiment × volatility, sentiment × uncertainty

### 5.5 Path Resolution (Critical)
All files use `__file__`-relative paths, NOT working-directory-relative:
```python
# sentiment_loader.py (3 parents up from app/backend/)
BASE_PATH = Path(__file__).resolve().parent.parent.parent / "stock-analysis" / "data" / "processed"

# evidence_agent.py (4 parents up from app/backend/hypothesis/)
DATA_ROOT = Path(__file__).resolve().parent.parent.parent.parent / "stock-analysis" / "data" / "processed"
```

## 6. Database (Supabase PostgreSQL)

### 6.1 Tables

```sql
-- Alert deduplication
sent_alerts (
    id          SERIAL PRIMARY KEY,
    alert_key   TEXT NOT NULL,
    alert_type  TEXT NOT NULL,
    ticker      TEXT,
    sent_at     TIMESTAMPTZ DEFAULT NOW()
)

-- Telegram subscribers
subscribers (
    id           SERIAL PRIMARY KEY,
    username     TEXT NOT NULL,
    telegram_id  TEXT,
    status       TEXT DEFAULT 'pending',  -- pending | approved | rejected
    requested_at TIMESTAMPTZ DEFAULT NOW(),
    approved_at  TIMESTAMPTZ
)

-- Prediction logging + outcome tracking
prediction_history (
    id               SERIAL PRIMARY KEY,
    ticker           TEXT NOT NULL,
    predicted_date   DATE NOT NULL,
    prediction       TEXT NOT NULL,       -- UP | DOWN
    confidence       REAL NOT NULL,
    actual_direction TEXT,                -- filled by evening scheduler
    actual_return    REAL,
    correct          INTEGER,             -- 0 | 1
    logged_at        TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(ticker, predicted_date)
)
```

### 6.2 Row Level Security (RLS)
- `sent_alerts`: service_role only (all operations)
- `subscribers`: anyone can INSERT, service_role only for SELECT/UPDATE/DELETE
- `prediction_history`: service_role only (all operations)

### 6.3 Connection

## 7. Authentication

### 7.1 Flow
```
User → /login page → supabase.auth.signInWithPassword()
     → Supabase issues JWT (ES256, 1hr expiry)
     → Frontend stores session via Supabase SDK (localStorage)
     → AdminPanel calls getSession() before every request
     → FastAPI auth.py verifies JWT via JWKS endpoint
     → require_admin() checks user_metadata.role === "admin"
```

### 7.2 JWT Verification (`auth.py`)
```python
# Fetches public keys from Supabase JWKS endpoint
# Supports ES256 (current default) with HS256 fallback
url = f"{SUPABASE_URL}/auth/v1/.well-known/jwks.json"
jwt.decode(token, jwks, algorithms=["ES256", "HS256"], options={"verify_aud": False})
```

### 7.3 Admin User Setup
- Created in Supabase Dashboard → Authentication → Users
- `raw_user_meta_data` set via SQL: `{"role": "admin", "email_verified": true}`
- Admin panel URL: `/ts-ops-7x9k` (obscured, not in nav)

## 8. Telegram Alert System

### 8.1 Scheduled Jobs
| Job | Schedule | Description |
|-----|----------|-------------|
| morning_brief | 9:30 AM ET daily | Predictions + top signals for all tickers |
| evening_brief | 4:15 PM ET daily | Actual outcomes vs predictions + accuracy |
| weekly_digest | 6:00 PM ET Sunday | Real accuracy stats from prediction_history |
| signal_watcher | Every 2 hours | Direction flips, sentiment spikes, litigation flags |

### 8.2 Subscriber Flow
```
User visits landing page → fills @username + #chat_id (from @userinfobot)
→ POST /subscribe
→ If telegram_id provided → auto-approved → welcome Telegram message sent
→ If no telegram_id → pending → admin approves via /ts-ops-7x9k
→ All approved subscribers receive all scheduled broadcasts via broadcast()
```

### 8.3 Bot Listener
- Polls for `/start` commands from Telegram
- Matches username to pending DB entries
- Auto-approves with chat_id from message object
- Runs in daemon thread with separate event loop (Linux-compatible fix)
- Conflict with localhost: expected — Railway runs the bot 24/7, don't run locally simultaneously

## 9. Hypothesis Engine (6 Agents)

```
User input: "Apple will drop 15% this quarter"
                    ↓
Agent 1: hypothesis_parser.py
  - Extracts ticker, target price, timeframe, implied return
  - Company name alias matching (Apple → AAPL)
  - Realism check: z-score of implied return vs historical distribution
  - Flags unrealistic hypotheses (z-score > 3)
                    ↓
Agent 2: market_collector.py
  - Fetches live feature row via feature_engineer.py
  - Returns current price, RSI, MA regime, momentum, volatility
                    ↓
Agent 3: evidence_agent.py
  - Computes base rates from merged_dataset.csv
  - "What % of historical X-day windows achieved >= implied return?"
  - Finds top 3 most similar historical setups (cosine similarity)
                    ↓
Agent 4: bear_agent.py          Agent 5: bull_agent.py
  - Tavily search (3 results)     - Tavily search (3 results)
  - Groq formats into JSON         - Groq formats into JSON
  - Returns sourced risk list      - Returns sourced catalyst list
                    ↓
Agent 6: synthesizer.py
  - Receives all 5 agent outputs
  - Groq llama-3.3-70b-versatile synthesizes full brief
  - feasibility_score (0-100):
      40pts base rate + 30pts technical alignment + 30pts realism
  - Returns structured JSON brief
                    ↓
POST /hypothesis/stream → SSE events per step → final brief
```

### 9.1 SSE Streaming Format
```
data: {"step": 1, "status": "running"}
data: {"step": 1, "status": "done", "ticker": "AAPL"}
data: {"step": 2, "status": "running"}
data: {"step": 2, "status": "done", "price": 189.3, "rsi": 44.2}
...
data: {"step": 0, "status": "complete", "brief": {...}}
```

## 12. Security Implementation

| Finding | Fix Applied |
|---------|------------|
| API keys in git | Never committed — confirmed via `git log --all --full-history -- "**/.env"` |
| Admin auth client-side | HTTP Basic → Supabase JWT ES256 on backend |
| Duplicate lifespan | Removed — single lifespan function confirmed |
| Auto-approve bypass | 24hr window + @userinfobot UX friction |
| No rate limiting | slowapi: 5/min hypothesis, 10/min explain, 3/min subscribe |
| Exception leaking | Generic "Internal server error." + server-side traceback logging |
| Frontend .env gitignored | Explicit entry in .gitignore |

## 16. Critical Rules (Learned from V1-V4)

1. **Always use `__file__`-relative paths** — never `os.getcwd()` or relative strings
2. **Always log ticker in every print statement** — prevents hardcoded ticker bugs
3. **`torch.load()` requires `weights_only=False`** — checkpoints contain non-tensor metadata
4. **Single Supabase client** — `src/lib/supabase.js` only, never instantiate in multiple files
5. **CPU-only PyTorch on Railway** — CUDA version = 6.9GB image = build failure
6. **`hypothesis_parser.py` not `parser.py`** — stdlib naming collision
7. **`broadcast()` not `send_message()`** in scheduler — send_message is admin-only
8. **ES256 not HS256** for Supabase JWT verification — fetch JWKS, don't hardcode algorithm
9. **`sympy==1.13.1` not `1.14.0`** — torch 2.6.0+cpu dependency conflict
10. **Never run local backend while Railway is live** — Telegram bot polling conflict
